Options Detail

Option 1: Hand-Crafted C99

Approach: Analyze PyTorch model → export weights via Python → implement all ShuffleNet V2 ops (conv2d, depthwise conv, BN, ReLU, channel shuffle, global avg pool, FC) in C99.

Strengths: Smallest code (1,660 lines), fully human-readable, maximum optimization control, int8 quantization support, NEON SIMD compatible, BN fusion possible, cache-friendly buffer management.

Weaknesses: Manual effort (~2 days), error-prone for complex architectures, requires deep understanding of ShuffleNet V2 internals.

Unique feature: Only option supporting int8 quantization (2.2 MB weights vs 8.7 MB float32).

Key files: shufflenet_v2.c (1,215 lines), shufflenet_v2.h (362 lines), main.c (445 lines), export_weights.py

Best for: Production deployment requiring maximum performance and minimum footprint.

Option 2: MATLAB Coder Support Package for PyTorch (R2026a)

Approach: loadPyTorchExportedProgram loads .pt2 model → MLIR/TOSA lowering → Embedded Coder generates C for ARM Cortex-A.

Strengths: Fully automated end-to-end, native embedded ARM target support, single monolithic C file (62,977 lines), numerically equivalent to PyTorch.

Weaknesses: New R2026a feature (less mature), requires Python environment with exact PyTorch version, large generated code.

Unique feature: Direct coder.hardware('ARM Cortex-A') support with Embedded Coder.

Key files: mInvoke_shufflenet.c (62,977 lines), generate_code_pytorch_coder.m

Best for: Rapid prototyping with guaranteed PyTorch numerical equivalence.

Option 3: importNetworkFromPyTorch → dlnetwork → MATLAB Coder CODEGEN FAILED

Approach: Import .pt2 model as dlnetwork using Deep Learning Toolbox → analyze for codegen → generate C with MATLAB Coder + Embedded Coder.

What worked: Import succeeds, MATLAB inference works perfectly (6.98 ms avg, bit-identical to Option 4), and the dlnetwork graph is inspectable.

What failed: C code generation fails because importNetworkFromPyTorch generates 30+ custom layers that inherit from nnet.layer.AutogeneratedFromPyTorch — a base class that does not support MATLAB Coder. Adding %#codegen pragmas is insufficient; the base class itself is incompatible.

Root cause: ShuffleNet V2's channel_shuffle operation (view + transpose + contiguous) and chunk (channel split) have no built-in MATLAB layer equivalents. The importer creates custom layers for:

  • view_view_* — 16 reshape layers for channel shuffle
  • chunk_* — 12 channel split layers
  • mean_* — 1 global average pooling via torch.mean()

Why Option 4 (ONNX) works but Option 3 doesn't: The ONNX import path generates custom layers from a different base class, and R2026a's GenerateCoderFiles=true creates a separate +coder/ package with codegen-compatible versions. The PyTorch import path has no such mechanism.

Key files: import_and_generate.m, predict_shufflenet.m, +shufflenet_exported/ (30 custom layers)

Verdict: Not viable for C code generation with ShuffleNet V2. Use Option 2 or 4 instead. Option 3 remains useful for MATLAB-only inference and transfer learning.

Option 4: PyTorch → ONNX → importNetworkFromONNX → MATLAB Coder

Approach: Export to ONNX (opset 17, legacy exporter) → import as dlnetwork → generate C. R2026a: auto-generated custom layers support codegen.

Strengths: Most portable format (ONNX), R2026a custom layer codegen confirmed working, broadest ecosystem support, GenerateCoderFiles=true generates +coder package.

Weaknesses: Two-step conversion (extra ONNX intermediate), more custom layers generated (Reshape, Slice, Gather, Squeeze ops), largest code output.

Unique feature: R2026a first release where auto-generated ONNX custom layers support C code generation out of the box.

Key files: export_to_onnx.py, import_onnx_and_generate.m, +shufflenet_v2/+coder/ (codegen layers)

Best for: Fallback when PyTorch import has issues, or when ONNX is the only available format.

Option 5: Rebuilt dlnetwork (codegen-compatible) COMPLETE

Approach: Import via importNetworkFromPyTorch, then replace all 30 AutogeneratedFromPyTorch custom layers with codegen-compatible equivalents.

Why Option 3 Fails: Three Codegen-Blocking Problems

Each autogenerated layer has three distinct problems, not just one:

  1. Incompatible base class: Layers inherit from nnet.layer.AutogeneratedFromPyTorch. MATLAB Coder rejects this class at the type-checking stage — no %#codegen pragma can fix it.
  2. Indirect dispatch: predict() doesn't do math directly — it calls tracedPyTorchFunction(obj, input, false, "predict"), an internal dispatch pattern Coder cannot trace through.
  3. Codegen-incompatible helper functions: The actual math lives in helpers (pyView, pyChunk, permuteToReversePyTorch) that use dynamic operations Coder can't resolve: dims(dlX) for runtime format queries, dlarray() constructors with format strings, struct/subsref for dynamic indexing, cell arrays with varargout, and a 90-line switch over 15 format strings.

Example call chain for channel shuffle: permuteToReversePyTorch (format switch) → tracedPyTorchFunction (dispatch) → pyView (cell/struct reshape) → pyTranspose (dynamic dim transform) → pyIdentity → another pyView. Six function calls, all codegen-incompatible.

Key Insight: Runtime Scaffolding vs. Compile-Time Math

All the autogenerated machinery — format-string switches, dimension permutation tables, dynamic cell/struct indexing, the tracedPyTorchFunction dispatch — exists to handle arbitrary PyTorch tensor layouts at runtime. But at code generation time, the tensor layout is known and fixed (SSCB for images). The actual math buried inside all that scaffolding is just:

  • Channel shuffle: 3 lines — reshape, permute, reshape
  • Channel split: 1 line — X(:,:,1:half,:) and X(:,:,half+1:C,:)

By writing replacement layers that implement only the known-layout math with static operations (all fully codegen-compatible) and inheriting from nnet.layer.Layer, all three problems are eliminated simultaneously. The replacement layers are simpler, faster, and produce bit-identical results because the mathematical operations are unchanged — only the runtime scaffolding is removed.

Issues Encountered & Fixes

Option 1: Hand-Crafted C

  • Channel shuffle complexity — ShuffleNet V2's channel shuffle (reshape+transpose+flatten) must be implemented efficiently without temporary allocation.
    Fix: In-place interleave with ping-pong buffer swap, avoiding extra memcpy.
  • Weight layout mismatch — PyTorch stores conv weights as [out_ch, in_ch, kH, kW], C implementation needs careful indexing.
    Fix: export_weights.py preserves PyTorch layout; C code indexes with matching stride calculations.

Option 2: PyTorch Coder (MLIR)

  • PyTorch version constraint — loadPyTorchExportedProgram requires specific PyTorch version matching the export.
    Fix: Use matching PyTorch version in Python venv, or re-export with compatible version.
  • Large monolithic output — Single 62,977-line C file with embedded weight constants.
    Impact: Compilation may be slow; consider splitting for large-scale projects.

Option 3: importNetworkFromPyTorch

  • RESOLVED: AutogeneratedFromPyTorch base class — importNetworkFromPyTorch generates 30+ custom layers (view_view, chunk, mean) that inherit from nnet.layer.AutogeneratedFromPyTorch. This base class does not support MATLAB Coder. Adding %#codegen pragmas is insufficient — the class itself is incompatible.
    Fix: Option 5 — Replace all 30 custom layers with codegen-compatible equivalents (ChannelShuffleLayer, ChannelSplitLayer, globalAveragePooling2dLayer). Same math, different class hierarchy.
  • NetworkLayer containers — Nested module structure requires expandLayers().
    Fix: net = expandLayers(net)
  • MATLAB inference works — Despite codegen failure, the imported dlnetwork runs correctly in MATLAB (6.98 ms avg, bit-identical to Option 4).

Option 5: Rebuilt dlnetwork

  • Key insight: replace base class, keep same operations — The 30 AutogeneratedFromPyTorch custom layers perform simple operations (reshape+permute for channel shuffle, indexing for channel split, mean for global avg pool). By reimplementing these as codegen-compatible ChannelShuffleLayer, ChannelSplitLayer, and globalAveragePooling2dLayer, the network becomes fully compatible with MATLAB Coder.
    Result: C codegen succeeded — 14 .c files, 16 .h files, 4,468 KB
  • Numerical equivalence confirmed — Max error 4.77e-06 vs original imported network, cosine similarity 1.0000000000, top-1 class match.
    Verdict: Mathematically identical to the original Option 3 dlnetwork, just with a different (codegen-compatible) class hierarchy.
  • Competitive C performance — 31.53 ms mean inference (5.53x vs Opt 1), only 7.8% slower than the fastest Option 2.
    Impact: Proves the PyTorch import path is viable for embedded C deployment when combined with layer replacement.

Option 4: ONNX Import

  • External data format — PyTorch dynamo ONNX exporter creates .onnx.data external weight file; MATLAB cannot import.
    Fix: Use legacy ONNX exporter (dynamo=False, opset 17).
  • Custom layer codegen — 17 auto-generated custom layers (Shape, Slice, Reshape, Gather, Squeeze, ReduceMean) with separate +coder package.
    Confirmed: R2026a GenerateCoderFiles=true creates codegen-compatible +coder package automatically.
  • Shape-dependent layers — ONNX Shape/Slice/Gather ops for dynamic reshape at channel shuffle generate multiple custom layers.
    Fix: Static input size eliminates runtime shape computation; codegen handles correctly.

Code Samples

Option 1: ShuffleNet V2 Channel Shuffle (C99)

/* Channel shuffle: interleave two halves of channels
 * Input:  C channels in CHW layout
 * Output: channels interleaved as [c0, c_half, c1, c_half+1, ...]
 * This is the key operation in ShuffleNet V2. */
static void channel_shuffle(const float *in, float *out,
                            int channels, int height, int width) {
    const int half = channels / 2;
    const int spatial = height * width;
    for (int c = 0; c < half; c++) {
        const float *src0 = in + c * spatial;
        const float *src1 = in + (half + c) * spatial;
        float *dst0 = out + (2 * c) * spatial;
        float *dst1 = out + (2 * c + 1) * spatial;
        memcpy(dst0, src0, spatial * sizeof(float));
        memcpy(dst1, src1, spatial * sizeof(float));
    }
}

Option 5: ChannelShuffleLayer (replaces 16 view_view layers)

classdef ChannelShuffleLayer < nnet.layer.Layer  % NOT AutogeneratedFromPyTorch
    %#codegen                                     % This pragma is what enables codegen
    properties
        NumGroups (1,1) double = 2
    end
    methods
        function Z = predict(layer, X)
            % Identical math to the AutogeneratedFromPyTorch view_view layer,
            % but inherits from nnet.layer.Layer -> MATLAB Coder accepts it
            G = layer.NumGroups;
            sz = size(X);
            H = sz(1); W = sz(2); C = sz(3);
            N = 1; if numel(sz) >= 4, N = sz(4); end
            CperG = C / G;

            Xr = reshape(X, [H, W, CperG, G, N]);    % [H,W,C/G,G,N]
            Xp = permute(Xr, [1, 2, 4, 3, 5]);       % swap G <-> C/G
            Z  = reshape(Xp, [H, W, C, N]);           % flatten back
        end
    end
end

Option 5: ChannelSplitLayer (replaces 13 chunk layers)

classdef ChannelSplitLayer < nnet.layer.Layer  % NOT AutogeneratedFromPyTorch
    %#codegen
    methods
        function layer = ChannelSplitLayer(name)
            layer.Name = name;
            layer.NumOutputs = 2;  % Two-output layer for ShuffleNet branch topology
        end
        function [Z1, Z2] = predict(~, X)
            % Split channels in half — same as torch.chunk(x, 2, dim=1)
            C = size(X, 3);
            half = C / 2;
            Z1 = X(:, :, 1:half, :);       % First half -> identity branch
            Z2 = X(:, :, half+1:C, :);     % Second half -> conv branch
        end
    end
end
